Using the Forge

Overview

Puppet Tasks enable you to easily automate ad hoc tasks and eliminate manual work across your infrastructure and applications. You can take your existing scripts, in any programming or scripting language and make them more robust and usable by converting them to a task.

Use Cases

Due to the ad-hoc nature of tasks, there are some use cases where these method is particularly appropriate such as:

  • Patching
  • Vulnerability management
  • Network device management
  • Maintenance tasks:
    • Rotate logs
    • Back ups
    • Restart system services

Components

Scripts

You can choose any script, written in any language to become a task in Puppet Enterprise, providing that your target node that run the script. This means you can leverage your existing investments and workflows when using Puppet tasks

Task Metadata

Tasks become a whole lot easier to use if they provide builtin explanations of what they do and how they can be used. This, and more, is what we can achieve with Task metadata.

Task metadata is a JSON file that accompanies a script, by having the same name as the script, but with a JSON file extension. The file contains descriptions for the Task itself, as well as for any of the Task’s parameters. It also allows you to easily control which parameters are mandatory vs optional, and what input values/types are allowed for each parameter. Task metadata also enables a number of more advanced capabilities, which you can find documented here.

Adding tasks to PE

Storage

Tasks will be recognized and available in Puppet Enterprise if they are contained within a module. A module is a container for puppet code automation in Puppet Enterprise. We’ll go more in depth on modules and their uses in the further sections.

Tasks can be added to a module directory within the site-modules directory of the control repository. Here is an example of basic module directory structure required for tasks: 

                        
  
                          Task name: maintenance::log_rotate
  
  
                          ├─ controlrepo/
                          └─ site-modules/
                              └─ maintenance/		        <-- Module Name
                                  └─ tasks/
                                      └─ log_rotate.sh    <-- Script
                                      └─ log_rotate.json  <-- Task
                                                              Metadata     
                                                                 
                                            
                        
                      
                        
                          Param($myparam1)
                          $systeminfo = systeminfo
                          Write-Output "System Info:" $systeminfo
                          Write-Output "Your custom parameter was: ${myparam1}"

Naming

Tasks are automatically namespaced with the format MODULENAME::TASKNAME.

You can see in the example above that the maintenance::log_rotate task is part of the maintenance module. Notice that the tasks script extension .sh has been dropped in the tasks name.

How it works

Once you've added your task to your control repo, you can then see your task via the Puppet Enterprise console running on Puppet Server. When you choose your task in PE and run it against your target node - two key things happen:

  1. The Puppet Server will copy the task content to remote targets agentlessly via SSH or WinRM or via the Puppet agent transport PCP.
  2. It will then run the task on the target and return the output to the Puppet console via the Puppet Server.

When using the PCP transport does not require credentials to run tasks on hosts. This is made possible because Puppet uses the PCP transport already for the Puppet agent and has full privileges to make changes on nodes. Actions can be limited to specific teams or people using RBAC to ensure these privileges cannot be abused.
